The AI assistant — how it works

The Workbench hands a language model the computed, cited reading and asks it to explain the result in plain language, narrate a Hermetic Codex of the Hour, or plan a historical working using the engine tools. Because the engine numbers are computed and passed in, the model narrates real results rather than inventing them.

Honest note The assistant narrates a historical, pseudoscientific tradition. A locked system prompt holds it to the site's framing: it describes Lilly's and the Picatrix's reasoning, it does not advise, predict, or present astrology as valid. Magical/talismanic content is historical only. See Sources & Science.

Two backends — Groq (free, the default) or Claude (agentic tools)

The assistant calls the model directly from your browser with your own key — nothing is proxied, and no key is ever bundled with the site:

  1. Groq (the default) — a genuinely free tier, no card needed: get a key at console.groq.com/keys. Fast open models (GPT-OSS 120B/20B, Qwen3). Streaming chat over the computed, cited reading; the grounded context is automatically slimmed to fit the free per-minute token cap.
  2. Claude (Anthropic) — the strongest experience, and the only backend with the agentic engine tools: get a key at platform.claude.com (default model claude-opus-4-8).
  3. Tick remember on this device to keep the key in this browser's localStorage for next time, or leave it off to re-enter the key each visit (it then lives only in the page for the session).
  4. Ask away — the same key works on the Workbench assistant and on every oracle page's diviner.
Privacy & CORS Requests go only to the chosen provider (api.groq.com or api.anthropic.com — the Anthropic call carries the anthropic-dangerous-direct-browser-access: true header that authorises a browser-origin request). Nothing is proxied through this site; the page is static. Your key is stored only where you choose (session memory, or this device if you tick "remember"). No network request fires on page load — every call happens on an explicit click.

Three ways to use it

  • Chat — ask anything about the reading (“why is the love election red?”, “which planet is strongest?”). With “let the model run the engine tools” ticked (Claude only), a tool-using model can compute fresh numbers itself — rank the aims, find the next good window, look up a mansion, even cast an oracle (geomancy, tarot, I Ching) — and narrate the real result.
  • 📜 Generate the Codex of this Hour — narrates the whole master report tool-by-tool in the grave, image-rich register of Hermes Trismegistus and the Picatrix (yet truthful): for each computation, what it measures, its meaning now, and how the tradition would best use it — with the honest caveat.
  • Plan a working (agentic) — the “conjure rain” box. Type a free-form aim (e.g. “when is the next best time to attempt to call rain, and how would the tradition do it?”). Claude maps it to the nearest catalogued operation, calls the engine tools (findNextElection, rankNow, talismanRecipe) to find the next favourable window, says what to check in the report, and lays out the historical procedure — described, not prescribed. Your requests are remembered for quick reuse.

How the grounding works

The assistant is built on three pure modules so the framing and facts can't drift:

  • assets/js/core/reading.js — fullReading composes the whole engine into one cited object.
  • assets/js/core/llm-context.js — buildContext() builds the locked honest-framing system prompt + the cited fact sheet; toAnthropicTools() exposes the engine's callable capabilities to Claude; runTool() safely runs the real engine for a tool call; buildCodexPrompt() and buildOperationPrompt() are the two task prompts.
  • assets/js/core/registry.js — the capability map the tool schema is derived from.

The diviner — the same assistant on the oracle pages

Each oracle page (Geomancy, Tarot, I Ching) carries an “Ask the diviner” panel built on the same transport and the same key. It speaks in the voice of a learned historian of the Western esoteric tradition — vivid about the figures, the cards and their rituals — yet it is held to the identical locked framing, and it is grounded in the computed cast (the shield, the spread, the hexagram): it interprets what was actually drawn, never inventing a figure or card. Every figure and card also carries a ✶ explain chip that pre-fills a focused question.

The local-model option (currently disabled)

The codebase also contains an offline path — a local model via Ollama or an in-browser WebLLM model — so the assistant can run with nothing leaving your machine. That option is turned off in the UI for now; the shared transport (assets/js/app/llm-core.js) already speaks the OpenAI-compatible protocol any local server exposes, so it can be re-enabled by adding the endpoint to the provider table. With Ollama you would run ollama serve with OLLAMA_ORIGINS set to allow this page; WebLLM needs WebGPU.
