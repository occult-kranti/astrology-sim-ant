Connect a local language model

The Workbench can hand its computed, cited reading to a language model running on your own machine and ask it to explain the result in plain language. The model only ever sees the engine's facts (already calculated, each with its source) — and nothing leaves the page: no cloud, no account, no telemetry.

Honest note The assistant narrates a historical, pseudoscientific tradition. It is held to the same framing as the rest of the site by a locked system prompt: it describes Lilly's and the Picatrix's reasoning, it does not advise, predict, or present astrology as valid. Magical/talismanic content is historical only. Astrology has no demonstrated predictive validity. See Sources & Science.

Why a local model fits this project

The whole site is static, offline-first and privacy-respecting. A local model keeps that promise: the engine computes the chart in your browser, builds a small grounded fact sheet (positions, dignities, the chart-health verdict, the Moon's condition, the election reasons, the talisman materials — each cited), and the model turns those facts into prose. Because the numbers are computed and passed in, the model narrates real results rather than inventing them.

Option A — Ollama (recommended)

  1. Install Ollama for your OS.
  2. Pull a model, e.g. a small instruct model: 
    ollama pull llama3.1
# or a lighter one:  ollama pull qwen2.5:7b   /   ollama pull phi3.5
  3. Allow this page to talk to Ollama (CORS). The Workbench runs in a browser, so Ollama must permit the page's origin. Set OLLAMA_ORIGINS before starting the server: 
    # Windows (PowerShell)
$env:OLLAMA_ORIGINS = "*"; ollama serve

# macOS / Linux
OLLAMA_ORIGINS="*" ollama serve
    (* allows any local page. To be stricter, list the exact origin you serve the site from, e.g. http://localhost:8003, or null if you open the files directly via file://.)
  4. On the Workbench, open Ask a local model, leave the backend on Ollama, click Detect / connect, pick your model, and ask.

Option B — WebLLM (fully in-browser)

No server at all: the model runs inside the browser tab using WebGPU (Chrome or Edge; recent GPU). Choose the WebLLM backend, click Detect / connect, and it downloads a quantised model on first use (often a few hundred MB to a couple of GB) and caches it. Heavier than Ollama, but zero install.

Two modes

  • Explain (default) — the model is given the grounded facts and narrates them. Works with any small model.
  • Tools (agentic) — tick “let the model run engine tools” and a tool-capable model (e.g. llama3.1, qwen2.5) may call the engine itself — cast a chart for another time, rank the election aims, find the next good window, look up a mansion — and narrate the real computed result. The tools come from the capability registry; the dispatcher refuses anything not on the list.

Things to ask

  • “Explain this chart in plain English.”
  • “What does the chart-health verdict mean here, and which testimonies drive it?”
  • “Why is the love election red right now? What would need to change?”
  • “Which planet is strongest and which weakest, and why, citing the dignities?”
  • “Find the next better hour for a wealth talisman.” (Tools mode)
Under the hood The grounding lives in assets/js/core/llm-context.js (pure, headless-tested): buildContext() builds the system prompt + cited facts, buildToolSchema() exposes the engine's callable capabilities, and runTool() safely dispatches a tool call to the real engine. The capability map is assets/js/core/registry.js; the one composed reading is assets/js/core/reading.js (fullReading).
